LocalMind 后端架构：秒杀、缓存、限流的设计与演进

发表于2026-05-26|更新于2026-05-26|项目实战
|浏览量:

背景

LocalMind 是一个本地生活优惠抢购平台，核心业务是限时秒杀。优惠券上架后库存有限、时间窗口短，瞬时流量集中。本文将梳理后端在缓存、数据一致性、秒杀链路、限流四个方面的设计思路和踩过的坑。

缓存体系：从击穿到热Key自动提升

为什么是两级而不是单级

单级 Redis 缓存的问题：网络 IO 本身也要 1-2ms，热点数据每秒几千次访问，累积的网络开销不可忽视。

选了 Caffeine（本地）+ Redis（分布式）两级：

击穿处理：逻辑过期 vs 互斥锁

缓存过期瞬间大量请求穿透到 DB，两种策略：

策略 做法 适用场景
互斥锁 拿到锁的线程回源，其余等待 一致性要求高的数据
逻辑过期 物理不过期，后台异步重建，未完成前返回旧值 允许短暂不一致的热点数据

对于商户详情这种允许短暂不一致的数据，用逻辑过期；订单状态用互斥锁。

热Key探测

两级缓存容量有限，”该热的不在本地”是常见问题。加了一个热Key探测模块：每次 Redis 命中时埋点计数，每 5 分钟采样 Top 20，对比两级命中率差异，自动将热点提升到 Caffeine。

缓存一致性：Canal + Kafka 异步失效

DB 更新后缓存未删，用户看到过期数据。方案选择：

方案 问题
更新 DB 后手动删缓存 代码侵入、可能忘记
延迟双删 第二次删除可能失败
Canal 监听 binlog 解耦、可靠，但引入额外组件

最终选了 Canal。MySQL 的每一次增删改都会产生 binlog，Canal 伪装成 MySQL slave 拉取解析，投递到 Kafka，消费者精准删除对应 Redis key。

消费失败处理：

  • 指数退避重试（10s → 30s → 90s）
  • 3 次失败转入死信队列，钉钉告警
  • 所有 key 强制 30 分钟 TTL 兜底——即使上面全挂，数据最多不一致半小时

秒杀链路：Lua 原子化 + 异步削峰

超卖问题

传统”查库存→判断→扣减”三步非原子，并发时两个请求同时读到库存=1，都去扣，就超卖了。

解决：把三步封装为一个 Lua 脚本，Redis 单线程执行，天然原子。

重复下单

用户快速双击产生两个请求。雪花算法生成全局唯一请求 ID（workId 取自 Redis 自增序列，避免时钟回拨），设为订单表唯一索引。重复请求在 DB 层直接被唯一键拦截。

超时关单

下单成功后把订单 ID 和过期时间戳写入 Redis ZSet，后台定时任务扫描到期订单，Redisson 分布式锁 + 事务保证关单和库存回滚的原子性。

异步削峰 + VIP 隔离

Lua 扣库存后不直接落库，投递消息到 Kafka，消费者异步写入。好处：秒杀瞬间的写压力被队列削平。

但单一消费者时普通用户频繁请求会挤占 VIP 处理资源。方案：创建两个独立线程池。

VIP 处理延迟从 320ms 降至 90ms。

限流与风控

自研 @RateLimiter 注解，基于 Redis ZSet 滑动窗口。支持三个维度独立配置：

  • 全局：保护整个接口
  • IP 维度：防爬虫
  • 用户维度：防单人刷单

连续触发限流 3 次，自动写入 Redis 黑名单封禁 5 分钟，从入口层拦截恶意流量。

总结

这些方案单独看都不新鲜，但放到一个真实业务项目里串联起来，才能真正理解——比如 Canal 加进来后 Kafka 消费者和秒杀消费者如何共处、两级缓存什么时候用逻辑过期什么时候用互斥锁、限流封禁怎样和业务错误码区分。这些细节才是实际写代码时真正要处理的问题。
