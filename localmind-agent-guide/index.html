用 LangChain4j 给 Spring Boot 项目加上 AI Agent：踩坑与思考

发表于2026-05-26|更新于2026-05-26|项目实战
|浏览量:

为什么不做独立 AI 项目，而是给业务项目”加”一个 Agent

去年做 LocalMind（一个本地生活优惠抢购平台）时，我发现一个现象：后台订单咨询量最大的不是复杂的客诉，而是三类重复问题——“订单到哪了””优惠券还能用吗””附近有什么店”。这其实就是个信息检索任务，不需要人来回答。

最初考虑过两个方案：

方案 优点 缺点
接入钉钉/企微机器人，写死关键词回复 简单 维护成本高，用户换个问法就失效
调 LLM API + 手写 Prompt 拼 SQL 灵活 每次迭代都要调 Prompt，难维护

最终选了第三条路：用 LangChain4j 的 AiServices 模式，把业务方法直接标注为 @Tool，让 LLM 自己决定调用哪个、怎么组合。

核心设计

LangChain4j 的模型是：定义一个接口 → 标注工具类 → 自动生成代理。关键在于工具类的设计。

LangChain4j 会自动把 @Tool 的方法名、参数类型、描述生成 OpenAI 兼容的 Function Definition，省去了手动拼 JSON Schema 的麻烦。这个细节在 DashScope SDK 里尤其重要——百炼的 Function Calling 格式和 OpenAI 不完全一致，LangChain4j 帮你做了一层适配。

踩坑

坑1：Session 隔离

最初直接在 Bean 初始化时构建一个全局 Agent 实例，结果用户 A 问”我的订单”，Agent 回忆起了用户 B 的对话。因为 MessageWindowChatMemory 是全局共享的。

解决：用 ConcurrentHashMap<String, CustomerAgent> 按 sessionId 隔离，每个 session 独立创建 Agent 实例和 Memory。会话过期后移除对应 entry。

代价是每个 session 一个 Agent 实例，内存吃紧时考虑 LRU 淘汰 + Redis 持久化对话历史。

坑2：System Prompt 时机

LangChain4j 的 ChatMemory 只在对话过程中自动追加 user/assistant 消息，System Prompt 需要手动加到第一条消息里。我在 Controller 层注入当前用户信息（昵称、VIP 等级）作为上下文，让 Agent 的回答更个性化。

坑3：工具返回值必须是自然语言

LLM 拿到工具返回结果后，要基于这些文本推理再输出回复。如果你返回的是 JSON 格式，LLM 理解成本高，容易出现”翻译错误”。所以我让每个工具方法直接返回自然语言——自己把数据库查出来的实体转成用户能读懂的文本。

效果

Agent 上线后，客服后台的重复咨询量明显下降——用户在对话框里问”我的 3 天前订的那个火锅券还能用吗”，Agent 自动调用查订单 + 查优惠券两个工具，30 秒内给出完整回复。代码不过 200 行。

但我不会说它”替代了人工客服”——它替代的是查找信息这一步，真正的投诉和纠纷还是转到人工。
